1. Create `lib/queries/reservations.ts` with server-side query functions:

`getCourtAvailability(date: string)`: Fetches all courts with their court_config (for the date's day type -- weekday/weekend), court_pricing, and existing reservations for the given date. Returns `CourtWithConfig[]` where each court has its schedule, pricing, and current reservations for that date.

The function should:
- Query `courts` joined with `locations` (for name, address)
- Query `court_config` filtered by day_type (determine from date: Sat/Sun = weekend, else weekday)
- Query `reservations` for the target date where status NOT IN ('cancelled', 'expired') -- include 'pending_payment' as occupied per Pitfall 4
- For pending_payment reservations, check if hold has expired: `created_at + hold_window < NOW()`. Expired holds should NOT count as occupied (on-demand check per RESEARCH discretion recommendation)
- Query `app_config` for hold duration
- Return structured data with courts, their configs, and per-slot availability
- IMPORTANT: Also compute a `courtAvailabilitySummary` for each court: count total slots and available slots for the day. Return `{ total: number, available: number }` per court so the card can display an availability badge.

`generateTimeSlots(config: CourtConfig, date: string, reservations: Reservation[])`: Pure function that generates an array of `TimeSlot` objects from the court config hours, checking each 1-hour slot against existing reservations. For open play slots, populate SpotInfo for spots 1-4. For full court slots, indicate if the court is booked or not.

2. Create `app/[locale]/(member)/reservations/actions.ts` with 'use server' directive:

`getAvailabilityAction(courtId: string, date: string)`: Server Action that TimeSlotGrid calls when the user switches date tabs. This allows client-side date switching without a full page reload.
- Authenticate user via supabase.auth.getUser()
- Call getCourtAvailability (filtered to the single courtId) for the requested date
- Return the time slots array and availability summary for that court+date
- This solves the RSC re-fetch problem: the page is an RSC that loads initial data, and subsequent date changes call this Server Action from the client.

3. Create `app/[locale]/(member)/reservations/page.tsx` as a Server Component:
- Authenticate user via supabase.auth.getUser()
- If not authenticated, redirect to /login
- Fetch user's membership status (to determine if they book free or need payment)
- Call getCourtAvailability for today's date
- Fetch app_config for advance booking windows
- Determine user's max advance booking days (member vs non-member)
- Pass all data as props to the client components (CourtCard for each court)
- Use next-intl useTranslations for page title and labels
- Page layout: title "Reserve a Court" / "Reserva una Cancha", then 3 court cards in a responsive grid (1 col mobile, 2 col md, 3 col lg)

4. Update `components/Navbar.tsx`: Add "Reservations" / "Reservas" link visible to all authenticated users (not gated by membership). Use the Reservations i18n namespace.

5. Add Reservations i18n namespace to `messages/en.json` and `messages/es.json`:
- Page title, court card labels, time slot labels, mode labels (Full Court / Open Play), spot availability text, date tab labels (Today, Tomorrow, etc.), booking button text, no availability text, pricing info text ("Free for members" / "$10 per session"), Google Maps link text, availability badge text ("X slots available", "Fully booked", "Limited availability")

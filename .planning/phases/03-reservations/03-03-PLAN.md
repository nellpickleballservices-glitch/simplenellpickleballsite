**createReservationAction(_prevState: unknown, formData: FormData)**:
Accepts: courtId, date, startTime, endTime, bookingMode, spotNumber (for open play), guestName (for VIP)

Flow:
1. Authenticate user via `supabase.auth.getUser()` -- redirect to /login if not authenticated
2. Fetch user's profile (for name snapshot + locale_pref for email)
3. **Pending payment block**: Query reservations where user_id = user.id AND status = 'pending_payment'. If any exist, return `{ error: 'pending_payment_block' }` (i18n key). Per CONTEXT.md: "Users with any pending payment reservation are BLOCKED from making new bookings."
4. **Membership check**: Query memberships for user_id with status 'active'. Determine isMember, isVip, memberLocationId.
5. **Advance booking window**: Get member_advance_booking_hours and non_member_advance_booking_hours from app_config. Calculate max booking datetime. If requested slot is beyond window, return `{ error: 'beyond_booking_window' }`.
6. **Basic tier location restriction** (RESV-07): If isMember and plan_type is 'basic', query the court's location_id. If court location_id !== membership.location_id, return `{ error: 'location_restricted' }`.
7. **VIP guest validation**: If guestName is provided, verify user isVip. Get vip_guest_limit from app_config. Non-VIP users cannot add guests.
8. **Determine payment status**: If isMember, payment_status = 'free', status = 'confirmed'. If not member, payment_status = 'pending_payment', status = 'pending_payment'.
9. **Determine price**: Query court_pricing for (courtId, bookingMode). If isMember, price_cents = 0. Else use court_pricing.price_cents.
10. **Insert reservation** (RESV-05 name snapshot): Insert into reservations with reservation_user_first_name and reservation_user_last_name from profile, booking_mode, spot_number, payment_status, guest_name, price_cents, starts_at/ends_at as TIMESTAMPTZ. For full_court mode, spot_number is NULL. The DB exclusion constraint handles race conditions -- catch unique_violation (23505) and return `{ error: 'slot_taken' }`.
11. **For full court mode**: Insert 4 rows (spots 1-4) in a single RPC call or use a Supabase function. Actually -- since the exclusion constraint for full_court does NOT use spot_number, a single row with spot_number=NULL is sufficient. The full_court exclusion constraint prevents any overlapping booking (including open_play) for the same court+time. Wait -- review the constraint: full_court constraint uses `WHERE booking_mode = 'full_court'`. This means it only prevents two full_court bookings from overlapping, but does NOT prevent an open_play booking during a full_court slot. The application must also check that no open_play reservations exist for the time range when booking full_court, and vice versa. Add a pre-insert check: query existing non-cancelled reservations for the court+time range regardless of mode. If any exist and the new booking is full_court, reject. If existing is full_court and new is open_play, reject.
12. **Send confirmation email** (NOTIF-01): Call sendConfirmationEmail with user's email (from auth), court name, formatted date, formatted time, and locale. This is fire-and-forget (await but don't fail the booking on email error).
13. Return `{ success: true, reservationId, needsPayment: !isMember }` so the UI can show payment options for non-members.

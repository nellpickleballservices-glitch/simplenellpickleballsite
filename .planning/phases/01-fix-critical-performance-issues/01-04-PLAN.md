**Create `lib/chat/rate-limit.ts`:**

Implement `checkRateLimit` following the pattern from RESEARCH.md:

- If no row: INSERT new row with count=1, window_start=now, return allowed=true

In `lib/queries/reservations.ts`, modify the `getCourtAvailability` function's `Promise.all` block.

Currently the reservations query fetches ALL reservations for the date regardless of whether a specific court is requested:
```typescript
supabase
  .from('reservations')
  .select('*')
  .gte('starts_at', `${date}T00:00:00`)
  .lt('starts_at', `${date}T23:59:59`)
  .not('status', 'in', '(cancelled,expired)')
```

Change it to conditionally add `.eq('court_id', courtId)` when courtId is provided:

```typescript
// Build reservations query — scope to specific court when courtId provided
let reservationsQueryBuilder = supabase
  .from('reservations')
  .select('*')
  .gte('starts_at', `${date}T00:00:00`)
  .lt('starts_at', `${date}T23:59:59`)
  .not('status', 'in', '(cancelled,expired)')

if (courtId) {
  reservationsQueryBuilder = reservationsQueryBuilder.eq('court_id', courtId)
}
```

Then use `reservationsQueryBuilder` in the `Promise.all` array instead of the inline query.

This change leverages the composite index `idx_reservations_court_starts` on `(court_id, starts_at)` created in Plan 01's migration.

Do NOT change any other part of the function. The court filtering in `.map()` (`reservations.filter(r => r.court_id === court.id)`) remains as a safety net for the multi-court case.
